# AfroPasso Agent Notes

## Project

AfroPasso is a static HTML/CSS/JavaScript site (no build step, no framework, no package.json) deployed to GitHub Pages from the repository root. It is a Polish-language educational guide to African and Afro-diasporic dances, starting with the kizomba family. Do not introduce a build step unless explicitly requested.

## Commands

There are no build, lint, or test commands. Verification before claiming completion:

```bash
node --check app.js                                # syntax-check the JS
node -e "JSON.parse(require('fs').readFileSync('data/dances.json'))"  # validate data
python3 -m http.server 8000                        # serve locally (fetch() requires HTTP, not file://)
```

If a local browser is available, inspect desktop and mobile layouts. Also check `git status -sb` and keep unrelated local artifacts (`.DS_Store`, `.npm-cache/`, `tmp/`, the loose research `.md`/`.txt` notes in the root) out of commits.

## Architecture

Single-page app with a hash router. All views live as `<section>` elements inside `index.html`; `app.js` shows/hides them.

- `index.html` — app shell and static markup for all views (home, Wszystkie tańce library, Społeczność/events, dance detail, timeline, compare, glossary).
- `app.js` — everything dynamic: global state at the top (`dancesData`, `filteredDances`, `eventsData`, `schoolsData`, `filteredEvents`, `currentRoute`), an `elements` object caching all DOM references (populated in `initializeElements()`), hash routing via `handleRoute()` (`#/`, `#/dances`, `#/events`, `#/dance/<slug>`, `#/timeline`, `#/compare`, `#/glossary`; the router listens on `hashchange` so plain-anchor links and back/forward route in all browsers), data loading, search/filtering, and card/detail/timeline/comparison/glossary/events rendering. Home (`#/`) shows a static 8-tile featured set (`renderFeaturedDances`) with a "Zobacz wszystkie tańce" button below it, then a static discovery-guide section (`.guide-panel`, hardcoded in index.html — 5 tiles linking to Społeczność/Etykieta/Korzenie/Porównaj/Glosariusz, no JS render function; shares `featured-motion.js`'s tile choreography, see that bullet). Search and filtering live on the Wszystkie tańce page (`#/dances`), which renders the full index via `renderDanceCards` and the shared `danceCardHTML()` template. The `#/events` "Społeczność" hub (`renderEventsPage`) shows featured event tiles (max 3), a month-grouped chronological list (past events auto-excluded), city/style/type filters whose options are derived from upcoming data, and a schools directory; the submission link uses the `EVENT_SUBMIT_FORM_URL` placeholder (replace with a real Google Form at launch). Events/schools load via a separate `loadEventsData()` try/catch so a failed fetch never breaks the dance guide. The glossary (`#/glossary`) is an expandable bento grid: `glossaryTerms` (in app.js, with `category`/`featured`/`danceSlug` fields) renders term-only tiles via `generateGlossary()`; clicking a tile FLIP-morphs it (Web Animations API) into the `#glossary-dialog` `<dialog>` in index.html with the full definition and an optional "Zobacz kartę tańca" link. Instant open/close under `prefers-reduced-motion`; the dialog needs `margin: auto` because the global `*` reset removes UA centering.
- `styles.css` — the full visual system and responsive layout.
- `lineage-motion.js` — progressive-enhancement scroll choreography for `#/timeline` ("Nić rodowodu"): scrubbed drawing of the map migration routes (full-bleed sticky prologue, `.lineage-map--scrub` adds the scroll room), an SVG thread drawn through the era nodes (geometry computed from node positions, rebuilt on ScrollTrigger `refreshInit`), chapter reveals, node activation, and the fixed era rail's active dot. Requires GSAP + ScrollTrigger (CDN, SRI); initializes lazily once the route is `#/timeline` and app.js has rendered `.lineage` (hashchange + MutationObserver). Static markup from app.js `renderTimeline()` is complete without it: alternating chapters around a CSS spine (`.lineage::before`), fully drawn map, "Nowy styl" birth pills (first historical appearance of each dance, image via `getFeaturedImage()`) linking to dance cards.
- `featured-motion.js` — progressive-enhancement tile motion ("Wejście na parkiet"): syncopated clip-path entrance (per-tile wipe directions + beat-pattern stagger), pointer-driven 3D tilt with inner-photo parallax and cursor-following sheen (`--mx`/`--my`, fine pointers only), neighbor focus dimming, and scrubbed column drift. A single reusable engine (`createTileMotion(config)`) drives two independent instances — the home featured-dances grid (`#featured-dances-container` / `.featured-tile`) and the discovery-guide grid (`#home-view .guide-grid` / `.guide-tile`) — each with its own tiles and teardown, so hover/neighbor-dimming in one grid never affects the other. Requires GSAP + ScrollTrigger; adds `.featured-motion-on` to whichever grid it initializes (CSS in styles.css neutralizes the old hover lift only in that mode, mirrored for both `.featured-tile`/`.guide-tile`). Each tile's photo lives in an inner `*-tile-media` layer so it can transform independently — `.featured-tile-media` is set inline per dance by `renderFeaturedDances()`, `.guide-tile-media` is set via a `--guide-image` custom property per `.guide-tile--*` class (static markup, see the `assets/` bullet for the not-yet-existing guide photos). Exits under `prefers-reduced-motion` / missing CDN, leaving both grids static with their CSS hover.
- `hero-motion.js` — progressive-enhancement motion layer for the home hero (deferred classic script; GSAP core comes from CDN with SRI, same precedent as the Firebase SDK). First-load entrance choreography (radial WebGL reveal of the hero photo growing outward from the dancing couple, word-by-word headline, staggered CTAs), then a hand-written WebGL fragment shader keeps the photo "alive" (simplex-noise breathing, cursor ripple, scroll parallax) plus gentle IntersectionObserver reveals on home sections. Strictly additive: exits early under `prefers-reduced-motion`, when the GSAP CDN fails, or without WebGL, leaving the pure-CSS page untouched; rendering pauses when the hero is off-screen or the tab hidden. The `.hero--gl` class swaps the CSS background photo for the canvas and moves readability scrims to `.hero::before` (styles.css).
- `data/dances.json` — canonical dance and timeline data, fetched at startup. Each dance has `id`/`slug`, `names`, `origin`, `classification`, `family`, `music`, Polish-language text fields (`*_pl`), `influences`, `differences_pl`, `controversies_pl`, and `videos`. Content changes usually mean editing this file, not JS.
- `data/events.json` + `data/schools.json` — events hub data (ship empty; real entries added via the curator tool). Event fields: `id`, `title`, `type` (`social`/`warsztaty`/`festiwal`), `styles` (dance slugs), `city`, `date_start`, optional `date_end`/`time`/`venue`/`price`/`image`, `url`, `featured`, `organizer`, `source`, optional non-rendered `confidence` (`single-source` marks an event published on one source, "to confirm").
- `tools/add-event.js` — dependency-free Node curator CLI that imports an event from a public URL's Open Graph tags (or manual entry), validates it, and inserts it chronologically into `data/events.json`. Its pure functions are unit-tested — run `node --test tools/add-event.test.js` (the directory form `node --test tools/` is broken on Node v26) before claiming tool changes complete. Publishing an event = run the tool, then git commit/push. Cover downloads go through `tools/cover-image.js` (WebP).
- **Research → verify → publish pipeline** (spec `docs/superpowers/specs/2026-07-21-event-research-pipeline-design.md`): an agent researches dance events, writes verified candidates to `data/candidates.json` (staging, never served), and a human promotes them. Non-blocking confidence: `verified` (≥2 independent sources agree on date+city + live link), `single-source` (1 source + live link, publishable but flagged), `unverified` (dead link / conflicting — not published).
  - `tools/research-sources.json` — registry of sources the research returns to, tagged `access` (`agent` = public HTML the agent reads / `human` = FB/IG/OnlyDance app, fed via paste-link), `role`, `quality`. Facebook is first-class quality but reached only indirectly (WebSearch index + aggregators + ticketing + human paste); there is no usable FB API for third-party event discovery.
  - `tools/promote-candidate.js` (+ `.test.js`) — dependency-free curator CLI: `--list` / `--promote <id>` / `--reject <id>` [`--force`]. Re-checks the link is live, downloads+converts the cover, reuses `add-event.js`'s `validateEvent`/`insertEventSorted`, inserts into `data/events.json`, removes the candidate. Blocks only `unverified` unless `--force`. Test: `node --test tools/promote-candidate.test.js`.
  - `tools/cover-image.js` (+ `.test.js`) — shared helper: download an event cover and convert to WebP (`cwebp -q 92 -m 6`), falling back to the original file if `cwebp` is absent. Used by both `promote-candidate.js` and `add-event.js`. On `#/events` the cover renders only on featured tiles (max 3) under a brand scrim, with a branded typographic fallback when there is no image; the month list stays text-only. Deferred (own future spec): cron auto-research + paid/featured monetization — `featured` stays editorial for now.
- `event-submit.js` — browser ES module (loaded via `<script type="module">` on top of classic `app.js`) rendering the on-site "Zgłoś wydarzenie" `<dialog>` on `#/events`. Lazy-loads the Firebase Web SDK from CDN and writes to the Firestore `submissions` collection (intake only). Public `FIREBASE_CONFIG` placeholder is filled by the owner after creating the project; a `window.__submitOverride` seam lets e2e drive the form without Firebase.
- `firestore.rules` + `firebase.json` — Firestore security rules (public create-only on `submissions` with field validation + honeypot; no public read). Deploy with `firebase deploy --only firestore:rules`.
- `tools/submission-core.js` — pure `validateSubmission` / `mapSubmissionToEvent` (reuse `add-event.js`); unit-tested via `node --test tools/submission-core.test.js`.
- `tools/pull-submissions.js` — curator CLI (`firebase-admin` + `tools/service-account.json`, both gitignored): `--list` / `--json` (agent-friendly) / `--approve <id>` / `--reject <id>`. Approving inserts into `data/events.json`; publishing is still `git commit && push`.
- `tools/` now has its own `package.json` (`firebase-admin`); run `cd tools && npm install` once. The served site stays dependency-free (Firebase Web SDK is loaded from CDN, not bundled).
- `assets/` — brand marks, hero, pillar icons, one image per dance in `assets/dances/`, curator-downloaded event images in `assets/events/`, and discovery-guide tile photos in `assets/guide/` (`spolecznosc.png`, `etykieta.png`, `korzenie.png`, `porownaj.png`, `glosariusz.png` — see `docs/superpowers/specs/2026-07-13-homepage-discovery-guide-design.md` section 7 for the generation prompts; until these exist the tiles fall back to their solid dark background color).

Use relative paths only (site must work under a `/afropasso/` subpath on GitHub Pages).

## Design constraints

`PRODUCT.md` and `DESIGN.md` define the product and visual direction — read them before UI work. Key hard rules:

- Palette lives in `DESIGN.md` frontmatter (warm paper, clay red, ochre, palm green, ink brown…). Never pure black or pure white.
- Forbidden: gradient text, glassmorphism, colored side-stripe card accents, generic icon grids, technical grid overlays as decoration, nightclub/neon styling, exoticizing imagery.
- Typography: serif display stack (Iowan Old Style/Palatino/Georgia), humanist sans body (Avenir/Segoe UI/system-ui).
- Tone: warm cultural documentary — respectful, educational, direct Polish copy without marketing inflation.
- Accessibility: target WCAG 2.1 AA — semantic HTML, keyboard navigation, visible focus states, reduced-motion support.

Brand: name **AfroPasso**, tagline **z pasji do tańców afrykańskich**, hero concept **Tańce, które łączą ludzi i historie**.
